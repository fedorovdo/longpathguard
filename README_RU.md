# LongPathGuard

LongPathGuard - небольшое audit-only приложение для Windows Server 2022. Оно наблюдает за новыми событиями файловой системы и помогает администратору находить слишком длинные полные пути и имена файлов или папок.

Приложение не удаляет, не перемещает и не переименовывает файлы. Существующее дерево каталогов проверяется только по явному запуску ручного сканирования.

## Скриншоты

Место для скриншотов подготовлено в `docs/screenshots/`. Туда можно добавить `dashboard.png`, `events.png`, `scan.png` и `settings.png` перед публикацией проекта.

## Возможности

- отслеживание `created`, `renamed` и, при включении, `modified` через watchdog;
- severity `ok`, `warning`, `danger`, `critical`, `long_name`, `critical_long_name`;
- хранение событий в SQLite и экспорт текущей выборки в CSV;
- dashboard со статистикой и последними событиями;
- ручной безопасный скан существующего дерева с лимитом объектов;
- получение владельца через Windows ACL/pywin32;
- интерфейс RU/EN на FastAPI и Jinja2;
- локальный запуск и установка службы через NSSM;
- Telegram и Email уведомления при заполненной конфигурации.

## Ограничения

- режим только Audit: карантина и автоматического исправления нет;
- встроенной авторизации в web UI нет;
- приложение не является драйвером файловой системы;
- ручной скан выполняется синхронно и ограничивается `max_scan_items`;
- NSSM не входит в репозиторий и не загружается автоматически.

## Требования

- Windows Server 2022 или совместимая Windows;
- 64-битный Python 3.12 в `PATH`;
- права чтения на наблюдаемую папку;
- права администратора только для установки Windows-службы;
- NSSM для service mode.

## Быстрый запуск

```powershell
Set-ExecutionPolicy -Scope Process Bypass
.\scripts\setup.ps1
.\scripts\run.ps1
```

`setup.ps1` проверяет Python 3.12, создает `.venv`, устанавливает зависимости, создает каталоги `config` и `data` и при отсутствии локального конфига копирует `config/config.example.yaml` в `config/config.yaml`. Существующий config не перезаписывается.

`run.ps1` использует готовую `.venv`, читает host/port из config и не переустанавливает зависимости при каждом запуске. Старый `run_dev.ps1` сохранен как совместимая обертка над setup и run.

По умолчанию UI доступен только локально:

```text
http://127.0.0.1:8787
```

## Первый запуск

В публичной конфигурации наблюдаемая папка намеренно не задана:

```yaml
watcher:
  root_path: ""
  excluded_paths: []
```

Это штатное состояние. Приложение запускается, но watcher и ручной скан не работают до выбора папки в разделе `Настройки`.

LongPathGuard принимает абсолютные локальные пути и UNC-пути. Внешние кавычки, переменные окружения и завершающий обратный слеш нормализуются. Относительные пути отклоняются. Приложение не создает введенную папку автоматически.

Примеры:

```text
D:\FileShare
"D:\Folder With Spaces\"
\\fileserver\share
%LPG_SHARE_ROOT%\Archive
```

Перед сохранением проверяются существование, тип объекта и возможность открыть каталог на чтение. Новый watcher запускается до остановки старого. Если проверка, запуск watcher или запись config завершились ошибкой, прежний config и прежний watcher продолжают работать.

## Основные настройки

Локальный config: `config/config.yaml`. Публичный пример: `config/config.example.yaml`.

```yaml
app:
  language: ru
  audit_mode: true
  host: 127.0.0.1
  port: 8787
events:
  store_ok_events: false
  store_modified_events: false
watcher:
  root_path: ""
  excluded_paths: []
scanner:
  max_scan_items: 10000
```

При `store_ok_events: false` обычные OK-события не пишутся в SQLite. При `store_modified_events: false` обычные modified-события пропускаются, но нарушения длины продолжают записываться.

Каталог `data` приложения всегда исключается внутренней проверкой, даже если он не указан в `excluded_paths`.

## Локальные и UNC-пути

Локальная папка, например `D:\FileShare`, доступна службе только при наличии NTFS-разрешений у ее учетной записи.

UNC-путь вида `\\server\share` работает только тогда, когда учетная запись Windows-службы `LongPathGuard` имеет права на SMB-ресурс и на файловую систему. `LocalSystem` обычно не следует использовать для доступа к удаленной сетевой папке.

Учетную запись службы можно настроить:

1. командой `nssm edit LongPathGuard`, вкладка `Log on`;
2. через `services.msc`, свойства `LongPathGuard`, вкладка `Вход в систему`.

После изменения учетной записи перезапустите службу. Логин и пароль службы не должны храниться в `config.yaml` или репозитории.

## Доступ из сети

Для доступа с других компьютеров можно изменить:

```yaml
app:
  host: 0.0.0.0
  port: 8787
```

Делайте это только в доверенной сети и после настройки Windows Firewall. В текущей версии нет сложной авторизации, поэтому не публикуйте порт напрямую в интернет.

## Ручное сканирование

Скан запускается только кнопкой на странице `Сканирование` и использует актуальный `root_path` из runtime-конфигурации. Исключения и `max_scan_items` применяются при каждом запуске. Ошибка доступа к отдельному подкаталогу записывается и не должна останавливать остальной обход.

При пустом или недоступном root_path кнопка отключена, а ложное событие в SQLite не создается.

## CSV-экспорт и язык

Страница `События` поддерживает фильтры severity, типа события, диапазона дат, поиска по пути и лимита. `Экспорт CSV` использует текущие фильтры.

RU/EN переключается в верхнем меню. Выбранный язык сохраняется в локальной конфигурации и таблице settings SQLite.

## Установка службы через NSSM

1. Скачайте NSSM вручную с официального сайта.
2. Положите `nssm.exe` в `scripts\tools\nssm.exe` или добавьте NSSM в `PATH`.
3. Запустите PowerShell от администратора:

```powershell
Set-ExecutionPolicy -Scope Process Bypass
.\scripts\install_service.ps1
.\scripts\start_service.ps1
```

Installer вызывает `setup.ps1`, проверяет отсутствие службы с таким именем, создает каталог `data`, использует host/port из config, включает автозапуск и restart on failure. Повторная установка поверх существующей службы блокируется.

Остановка и запуск:

```powershell
.\scripts\stop_service.ps1
.\scripts\start_service.ps1
```

Удаление:

```powershell
.\scripts\uninstall_service.ps1
```

Скрипты не загружают NSSM и не сохраняют учетные данные службы.

## Файлы, данные и логи

- конфигурация: `config/config.yaml`;
- пример: `config/config.example.yaml`;
- SQLite: `data/longpathguard.db`;
- лог приложения: `data/longpathguard.log`;
- stdout/stderr службы: `data/service.out.log`, `data/service.err.log`.

Локальный config, база и логи исключены из Git.

## Резервное копирование

Для согласованной копии сначала остановите службу, затем скопируйте `config/config.yaml` и каталог `data` в защищенное место:

```powershell
.\scripts\stop_service.ps1
Copy-Item .\config\config.yaml E:\Backups\LongPathGuard\config.yaml
Copy-Item .\data E:\Backups\LongPathGuard\data -Recurse
.\scripts\start_service.ps1
```

Путь назначения приведен как пример. Не публикуйте backup в Git.

## Обновление

1. Остановите службу.
2. Сделайте резервную копию config и data.
3. Обновите файлы приложения, не заменяя локальные `config/config.yaml` и `data`.
4. Выполните `.\scripts\setup.ps1` для обновления зависимостей.
5. Запустите тесты и службу.

```powershell
.\scripts\stop_service.ps1
.\scripts\setup.ps1
.\.venv\Scripts\python.exe -m pytest
.\scripts\start_service.ps1
```

## Безопасность

- оставляйте `127.0.0.1` по умолчанию, если удаленный доступ не требуется;
- ограничивайте порт 8787 Windows Firewall;
- запускайте службу с минимальными правами чтения;
- не храните пароли службы, SMTP или Telegram credentials в Git;
- защищайте backup SQLite и config;
- помните, что Audit mode не блокирует создание длинного пути и не исправляет его автоматически.

## Troubleshooting

- `Путь не указан`: задайте абсолютную папку в `Настройки`.
- `Требуется абсолютный путь`: используйте `D:\Share` или `\\server\share`, не относительный путь.
- `Папка не существует`: проверьте опечатку, подключение диска или доступность SMB.
- `Объект не является папкой`: выберите каталог, а не файл.
- `Нет доступа`: проверьте NTFS/SMB права учетной записи, под которой запущен процесс или служба.
- `Watcher не запустился`: смотрите `data/longpathguard.log` и service stderr.
- `nssm.exe не найден`: положите файл в `scripts\tools\nssm.exe`; он намеренно не включен в Git.
- UI не открывается удаленно: проверьте host, firewall и доверенность сети.
- Порт занят: измените `app.port` или остановите другой процесс, использующий 8787.
